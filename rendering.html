Rendering System

Overview

The LCE rendering pipeline is coordinated by three core classes. GameRenderer serves as the top-level render coordinator: it configures the camera, builds the light texture, and delegates world drawing to LevelRenderer. LevelRenderer is responsible for chunk-based world rendering, entity drawing, sky and cloud passes, and particle effects. Actual geometry submission flows through the Tesselator, a thread-local singleton that assembles vertex buffers with position, texture coordinate, color, and normal data before flushing them to the graphics API.

A key design consideration is split-screen multiplayer. The engine supports up to 4 independent viewports, each with its own camera, FOV, light texture, level reference, chunk array, and tile renderer instance. Arrays indexed by player slot (e.g. fov[4], level[4], chunks[4]) appear throughout the renderer classes to facilitate this.

Visibility determination uses a Culler interface for frustum culling. Before any chunk or entity is drawn, it is tested against the active culler to discard geometry that falls outside the current viewport's view frustum, keeping draw calls within budget on console hardware.

Tesselator

Tesselator
Minecraft.Client/Tesselator.h

Thread-local singleton for submitting geometry to the graphics API. The Tesselator manages a growable vertex buffer and tracks per-vertex attributes (position, texture coordinates, color, normals). All block, entity, and particle rendering ultimately flows through this class. Each rendering thread maintains its own Tesselator instance to avoid synchronization overhead.

TileRenderer

TileRenderer
Minecraft.Client/TileRenderer.h

Renders block (tile) faces into the Tesselator. One TileRenderer instance exists per player viewport to support split-screen rendering. Handles all block render shapes including standard cubes, cross-shaped plants, slabs, stairs, fences, doors, redstone wire, torches, rails, and other specialized geometries. Each shape variant has a dedicated render method that computes the correct UVs and vertex positions based on the block's metadata and surrounding neighbors.

EntityRenderDispatcher

EntityRenderDispatcher
Minecraft.Client/EntityRenderDispatcher.h

Central dispatch point for entity rendering. Maintains a registry mapping entity types to their corresponding EntityRenderer subclass. When LevelRenderer::renderEntities iterates the entity list, each entity is passed to the dispatcher, which looks up the appropriate renderer and invokes its render method with the correct position, rotation, and partial-tick interpolation values. Also manages the global camera position used by all entity renderers for distance-based LOD and facing calculations.

Particle System

ParticleEngine
Minecraft.Client/ParticleEngine.h

Manages all active particle instances in the world. Referenced by LevelRenderer, the particle engine is responsible for ticking, rendering, and culling the particle pool. Particles are grouped by render layer (normal, translucent, lit) and drawn in batch after the main world pass. The engine handles particle creation from LevelRenderer::addParticle calls, lifetime management, and disposal of expired particles.

Particle
Minecraft.Client/Particle.h

Abstract base class for all particle types. Each particle tracks its world position (x, y, z), velocity (xd, yd, zd), remaining lifetime, texture region, RGBA color, and billboard size. Subclasses override tick() to implement custom motion (gravity, drag, oscillation) and render() to customize the visual output (texture animation, color fade, scaling).

Texture System

LCE uses a texture management stack that handles loading, caching, atlas stitching, and runtime texture pack switching. The system is designed to support the default built-in assets as well as downloadable content (DLC) texture packs on console platforms.

TextureManager
Minecraft.Client/TextureManager.h

Central texture resource manager. Loads image files from the active texture pack, uploads them to the GPU, and manages the resulting texture handles. Provides methods for binding textures by name or ID, and handles animated texture ticking (e.g. water, lava, fire). Also responsible for reloading all textures when the active texture pack changes.

TexturePack
Minecraft.Client/TexturePack.h

Abstract interface for a texture pack. Provides methods to open image resources by path, query available textures, and read pack metadata. Concrete implementations handle different resource sources (built-in, file system, DLC).

StitchedTexture
Minecraft.Client/StitchedTexture.h

Represents a texture atlas built by stitching together multiple individual block or item textures into a single large GPU texture. Stores the UV mapping table that maps each source texture name to its region within the atlas, enabling efficient batched rendering with minimal texture bind calls.

Stitcher
Minecraft.Client/Stitcher.h

Atlas packing algorithm. Takes a set of individual textures with varying dimensions and packs them into a power-of-two atlas texture with minimal wasted space. Uses a bin-packing strategy to arrange sprites, then outputs the final atlas image and a UV coordinate lookup table consumed by StitchedTexture.
