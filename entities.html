Entity System

The entity system is the backbone of all game objects in Minecraft LCE. Every mob, player, projectile, item drop, vehicle, and special effect in the world is an entity. This page documents the full class hierarchy, the AI goal system, pathfinding infrastructure, control systems, and every concrete mob type in the codebase.

Entity Hierarchy

All game objects derive from a single Entity base class. The hierarchy branches into living mobs (with health, AI, and navigation), projectiles, vehicles, hanging decorations, boss entities, and special-purpose types. The full inheritance tree is shown below.

Entity (enable_shared_from_this<Entity>)
├── Mob
│   ├── PathfinderMob
│   │   ├── AgableMob
│   │   │   ├── Animal (+ Creature)
│   │   │   │   ├── Cow, Pig, Sheep, Chicken
│   │   │   │   ├── Wolf, Ocelot (tameable)
│   │   │   │   └── Bat
│   │   │   └── Villager
│   │   └── Monster (+ Enemy)
│   │       ├── Creeper, Spider, CaveSpider
│   │       ├── Zombie, PigZombie, Skeleton
│   │       ├── Blaze, Slime, Ghast, Silverfish
│   │       └── Enderman
│   └── Player (+ CommandSender)
├── Arrow, Boat, ItemEntity
├── HangingEntity (Painting, ItemFrame)
├── GlobalEntity (Lightning)
├── BossMob, BossMobPart
└── PistonPieceEntity

The hierarchy uses multiple inheritance in several places: Animal also extends the Creature interface, Monster extends the Enemy interface, and Player extends the CommandSender interface. These mixins provide trait-based classification without adding state.

Entity Base Class

Entity Base Class

File: Minecraft.World/Entity.h

Extends: enable_shared_from_this<Entity>

Base class for all game objects with position, physics, collision, and serialization. Every object that exists in the world — from players and mobs to dropped items and arrows — inherits from this class. It provides the fundamental tick loop, movement and collision detection, damage handling, serialization for save/load, and synched entity data for client-server replication.

Position & Movement Members

Member Type Description
x, y, z double Current world position of the entity. Updated every tick by the movement system.
xo, yo, zo double Previous tick position. Used for interpolation during rendering to produce smooth visual movement between ticks.
xd, yd, zd double Velocity (delta) on each axis. Applied to position during move() and modified by gravity, friction, and knockback.
yRot, xRot float Current rotation angles. yRot is the yaw (horizontal rotation, 0–360) and xRot is the pitch (vertical look angle, -90 to +90).
yRotO, xRotO float Previous tick rotation values used for interpolation, matching the old-position pattern.

Physics Members

Member Type Description
bb AABB* Axis-aligned bounding box used for collision detection. Defines the entity's physical volume in world space.
bbWidth float Width of the bounding box on the X and Z axes (entities are symmetrical on the horizontal plane).
bbHeight float Height of the bounding box on the Y axis.
onGround bool Whether the entity is currently standing on a solid surface. Controls jump eligibility and fall damage calculation.
horizontalCollision bool Set to true when the entity collides with a block on the X or Z axis during move().
verticalCollision bool Set to true when the entity collides with a block on the Y axis (ceiling or floor contact).
fallDistance float Accumulated distance fallen since the entity last touched ground. Used to calculate fall damage on landing.
walkDist, walkDistO float Total distance walked (current and previous tick). Drives walk animation cycling in the renderer.

Status Members

Member Type Description
entityId int Unique numeric identifier assigned when the entity is added to the level. Used in networking packets to reference entities.
level Level* Pointer to the level (world) this entity belongs to. Provides access to block data, other entities, and world state.
removed bool Marks the entity for removal. When true, the entity will be cleaned up at the end of the current tick cycle.
tickCount int Number of ticks this entity has existed. Incremented every baseTick() call; used for age-based behaviors and despawning.
invulnerableTime int Remaining invulnerability frames after being hit. Prevents damage stacking from rapid successive hits.
flameTime int Remaining ticks the entity is on fire. Decrements each tick; applies fire damage periodically while positive.
entityData shared_ptr<SynchedEntityData> Synchronized data container for client-server replication. Stores data entries (health, flags, variant IDs) that are automatically synced over the network.
rider weak_ptr<Entity> The entity currently riding on top of this entity (e.g., a player riding a pig). Weak pointer to avoid circular references.
riding shared_ptr<Entity> The entity this entity is currently riding. Shared pointer because the ridden entity must stay alive while being ridden.

Key Methods

Method Signature Description
tick void tick() Main update method called every game tick (20 times per second). Subclasses override this to implement behavior.
baseTick void baseTick() Core tick logic shared by all entities: increments tickCount, handles fire damage, water/lava checks, and fall damage reset.
move void move(double xa, double ya, double za, bool noEntityCubes) Moves the entity by the given deltas, performing full AABB collision detection against blocks and optionally other entities. Updates onGround, collision flags, and fallDistance.
setPos void setPos(double x, double y, double z) Teleports the entity to the given coordinates. Updates the bounding box and re-registers the entity in the level's spatial grid.
turn void turn(float xo, float yo) Adjusts rotation by the given deltas. Clamps pitch to -90..+90 range and wraps yaw to 0..360.
hurt bool hurt(DamageSource*, int damage) Applies damage from a source. Returns true if damage was actually dealt (respects invulnerability frames). Triggers knockback and hurt animation.
die void die(DamageSource*) Called when the entity's health reaches zero. Handles death effects, loot drops, experience orbs, and marks the entity as removed.
isAlive bool isAlive() Returns true if the entity has not been removed and (for mobs) has health greater than zero.
isInWater bool isInWater() Checks whether the entity's bounding box overlaps with any water blocks.
isInLava bool isInLava() Checks whether the entity's bounding box overlaps with any lava blocks.
isInWaterOrRain bool isInWaterOrRain() Returns true if the entity is submerged in water or exposed to rain at its current position.
getLightColor int getLightColor(float partialTick) Returns the packed light color value at the entity's position, used for entity rendering brightness.
getBrightness float getBrightness(float partialTick) Returns the brightness level (0.0–1.0) at the entity's position for lighting calculations.
save void save(CompoundTag*) Serializes the entity's state into an NBT compound tag for world saving. Writes position, velocity, rotation, fire, and custom subclass data.
load void load(CompoundTag*) Deserializes entity state from an NBT compound tag when loading a saved world. Restores all serialized fields.
interact bool interact(shared_ptr<Player>) Called when a player right-clicks on this entity. Returns true if an interaction occurred (e.g., mounting, feeding, trading).
getCollideBox AABB* getCollideBox() Returns the entity's collision box used for entity-vs-entity collision. Can be nullptr for non-collidable entities.
distanceTo double distanceTo(Entity) Calculates the Euclidean distance between this entity and another entity.

Mob Class

Mob Living Entity

File: Minecraft.World/Mob.h

Extends: Entity

Living entity with health, AI system, navigation, and controls. This is the base for all entities that can take damage, die, have status effects, and exhibit autonomous behavior. The Mob class introduces the AI goal system (GoalSelector), pathfinding navigation, and the control system trio (look, move, jump) that subclasses configure to produce specific mob behavior.

Health & Combat Members

Member Type Description
health int Current health points. When this reaches zero, die() is called. Different mob types have different max health values.
lastHealth int Health value from the previous tick. Used to detect health changes and trigger damage flash effects on the client.
deathTime int Ticks since death. Incremented during the death animation; when it reaches 20, the entity is removed from the world.
hurtTime int Remaining ticks of the hurt animation (red flash). Set when the mob takes damage, decrements each tick.
hurtDuration int Total duration of the current hurt animation in ticks. Used to calculate the intensity of the visual hurt effect.
arrowCount int Number of arrows visually stuck in the mob. Decremented over time; purely cosmetic after the initial hit.
target shared_ptr<Mob> The mob's current attack target, set by the targeting goal system. AI combat goals use this to know whom to attack.
lastHurtByPlayer shared_ptr<Player> The last player that dealt damage to this mob. Used to determine experience and loot drop eligibility.
activeEffects map<int, MobEffectInstance*> Map of active status effects keyed by effect ID. Includes potion effects like speed, strength, poison, and regeneration.

AI System Members

Member Type Description
goalSelector GoalSelector Primary goal manager. Holds all behavior goals (movement, interaction, etc.) and evaluates them each tick by priority.
targetSelector GoalSelector Targeting goal manager. Separate from goalSelector so targeting decisions run independently from behavior goals.
navigation PathNavigation* Pathfinding navigation system. Manages path creation and following, allowing the mob to move toward goals while avoiding obstacles.
lookControl LookControl* Controls where the mob is looking. Goals set a wanted look direction and the control smoothly interpolates the head rotation.
moveControl MoveControl* Controls mob movement. Goals set a wanted position and speed; the control applies the appropriate movement forces each tick.
jumpControl JumpControl* Controls jumping. Goals can request a jump and the control will execute it on the next tick when conditions allow.
bodyControl BodyControl* Controls body posture and rotation. Prevents the body from rotating to unrealistic angles relative to the head.
sensing Sensing* Vision and awareness system. Caches line-of-sight checks per tick so multiple goals can query visibility efficiently.

Key Methods

Method Signature Description
aiStep void aiStep() Per-tick AI update. Runs goal selectors, updates navigation, applies controls, handles status effects, and manages despawn logic.
serverAiStep void serverAiStep() Server-side AI logic. Called within aiStep() only on the server; runs the goal selectors and navigation ticking.
hurt bool hurt(DamageSource*, int) Overrides Entity::hurt to add armor reduction, potion effect modifiers, and mob-specific damage responses.
doHurtTarget bool doHurtTarget(Entity*) Attacks a target entity. Applies this mob's attack damage, enchantment effects, and knockback to the target.
die void die(DamageSource*) Overrides Entity::die to drop mob-specific loot, award experience, play death sound, and run the death animation.
knockback void knockback(Entity*, int, double, double) Applies knockback force from a hit. The strength and direction are calculated from the attacker's position and the knockback amount.
getTarget shared_ptr<Mob> getTarget() Returns the mob's current attack target, or nullptr if no target is set.
setTarget void setTarget(shared_ptr<Mob>) Sets the mob's attack target. Used by targeting goals to direct combat behavior.
getMaxHealth int getMaxHealth() Returns the maximum health for this mob type. Subclasses override to define their specific max health value.
addEffect void addEffect(MobEffectInstance*) Applies a status effect (potion effect) to the mob. If an effect of the same type already exists, it is replaced if the new one is stronger or longer.
removeEffect void removeEffect(int) Removes a status effect by its effect ID.
canSee bool canSee(Entity*) Performs a line-of-sight raycast to determine if this mob can see the given entity. Results are cached in the Sensing system.

PathfinderMob & AgableMob

These intermediate classes sit between Mob and the concrete mob types, adding pathfinding-driven behavior and age/breeding support respectively.

PathfinderMob Abstract

Extends: Mob

A mob that can pathfind through the world. Adds an explicit attack target and random stroll capabilities. Most hostile and passive mobs extend this class rather than Mob directly.

Members: Path* path (current navigation path), attackTarget (shared pointer to the targeted entity).

Key Methods:

  • findRandomStrollLocation() — Selects a random walkable position near the mob for idle wandering behavior.
  • isPathFinding() — Returns true if the mob is currently following an active path.
  • setPath() — Assigns a new path for the navigation system to follow.
AgableMob Abstract

Extends: PathfinderMob

A mob that has an age and can breed. Negative age values indicate a baby; positive values indicate the cooldown timer after breeding. Age is stored as synched entity data so baby/adult state is replicated to clients.

Members: age (synched int, negative = baby, zero = adult, positive = breeding cooldown).

Key Methods:

  • getAge() — Returns the current age value.
  • setAge(int) — Sets the age. Negative values create a baby mob with smaller bounding box and rendering scale.
  • isBaby() — Returns true if age is negative.
  • getBreedOffspring() — Creates a new mob of the same type as a baby. Called during successful breeding.

Animal Class

Animal Passive Mob Base

Extends: AgableMob, Creature

Base class for all passive breedable animals. The Creature mixin marks this entity type as a creature for spawning rules (spawns on grass in light). Animals can be lured with food items, enter "love mode" for breeding, and produce offspring.

Members:

  • loveTime — Ticks remaining in "love mode" (shown by heart particles). Decrements each tick; breeding occurs when two nearby animals are both in love mode.
  • loveCause — The player who triggered love mode by feeding the animal. Used to award breeding experience to the correct player.

Key Methods:

  • isInLove() — Returns true if loveTime > 0, meaning the animal is ready to breed.
  • setInLove() — Enters love mode, setting loveTime and spawning heart particles.
  • breedWith(Animal*) — Executes breeding with another animal: spawns a baby, awards experience, and resets both animals' love timers.
  • canMate(Animal*) — Returns true if both animals are the same type, both in love mode, and neither is a baby.

Monster Class

Monster Hostile Mob Base

Extends: PathfinderMob, Enemy

Base class for all hostile mobs. The Enemy mixin marks this entity type as hostile for spawning rules (spawns in darkness) and changes how players interact with it (auto-targeting). Monsters attack players and other non-hostile mobs on sight, and only spawn in sufficiently dark areas.

Members:

  • attackDamage — Base melee attack damage dealt to targets. Modified by difficulty level and any held weapon enchantments.

Key Methods:

  • findAttackTarget() — Scans for the nearest valid attack target (typically the closest player within range). Called by targeting goals.
  • isDarkEnoughToSpawn() — Checks whether the light level at the mob's position is low enough for hostile mob spawning (light level 7 or below).

Player Class

Player Player Entity

Extends: Mob, CommandSender

The player entity. Unlike all other mobs, the player is driven by real user input rather than AI goals. The CommandSender mixin enables the player to execute chat commands. The Player class manages the player's inventory, ender chest storage, game mode abilities, food/hunger system, experience progression, and all player-specific interactions like sleeping, crafting, and container access.

Members:

  • inventory — The player's main inventory containing hotbar, armor slots, and general storage. Manages item stacking, equipment, and slot selection.
  • enderChestInventory (PlayerEnderChestContainer) — Per-player ender chest storage. Contents are unique to each player and persist across all ender chest blocks.
  • abilities (Abilities) — Player abilities struct controlling game mode flags: mayfly, flying, instabuild (creative mode), invulnerable, mayBuild, and walkSpeed/flySpeed.
  • foodData (FoodData) — Hunger and saturation system. Tracks foodLevel (0–20), saturationLevel, and exhaustionLevel. Controls natural regeneration and sprint eligibility.
  • experience — The player's experience points and level, used for enchanting and anvil operations.

Key Methods:

  • interact() — Handles right-click interactions with entities (trading with villagers, feeding animals, mounting horses).
  • attack() — Executes a melee attack on the target entity, applying weapon damage, enchantment effects, critical hits, and knockback.
  • startSleepInBed() — Attempts to sleep in a bed. Checks for nearby hostile mobs, daytime restrictions, and bed occupancy before allowing sleep.
  • respawn() — Resets the player after death: restores health, clears inventory (in survival), and teleports to the spawn point or bed location.
  • openContainer() — Opens a container GUI (chest, furnace, crafting table, etc.) and establishes the container synchronization channel with the client.

AI Goal System

The AI goal system drives all autonomous mob behavior through a priority-based goal manager called GoalSelector. Each mob registers a set of goals at construction time, and every tick the selector evaluates which goals should be running.

GoalSelector AI Core

Priority-based goal manager. Goals are registered with an integer priority (lower number = higher priority). Each tick, the selector iterates through all registered goals and determines which ones should be active based on the following lifecycle:

  1. canUse() — Called on inactive goals to check if they can start. Returns true if preconditions are met (e.g., target in range, correct time of day, mob not already doing something more important).
  2. start() — Called once when the goal transitions from inactive to active. Performs initialization (e.g., set navigation target, begin animation, cache references).
  3. tick() — Called every tick while the goal is active. Performs the ongoing behavior (e.g., move toward target, track look direction, update timer).
  4. canContinueToUse() — Called on active goals to check if they should keep running. Returns false to stop the goal (e.g., target died, reached destination, timer expired).
  5. stop() — Called once when the goal transitions from active to inactive. Performs cleanup (e.g., clear navigation, reset animation state).

When multiple goals can run simultaneously, higher-priority goals (lower number) take precedence and can preempt lower-priority goals. Goals also declare mutex flags to prevent incompatible goals from running at the same time (e.g., two movement goals cannot both control the mob's legs).

The codebase contains 41 distinct AI goals, listed below in categorized tables.

Movement Goals (7)
Goal Description
RandomStrollGoal Wanders to a random nearby position when idle. Calls findRandomStrollLocation() to pick a walkable destination within a configurable radius. One of the lowest-priority goals, providing default idle movement.
MoveTowardsTargetGoal Moves the mob toward its current attack target. Used by melee mobs (iron golems, zombies) to close distance before attacking. Continuously updates the path as the target moves.
MoveTowardsRestrictionGoal Moves the mob back toward its home position (spawn point or village center). Activates when the mob has strayed too far from its restriction area. Used by village-bound mobs like iron golems.
MoveThroughVillageGoal Guides mobs along village paths, navigating through doors and between buildings. Selects random village waypoints and creates paths through the village infrastructure.
MoveIndoorsGoal Moves mobs indoors (into village houses) during rain or nighttime. The mob seeks the nearest door and navigates inside. Used primarily by villagers for shelter-seeking behavior.
FloatGoal Causes the mob to swim upward when submerged in water. Without this goal, most mobs sink and drown. Sets the jump flag when the mob's head is underwater. Typically the highest-priority goal (priority 0).
FollowParentGoal Baby animals follow their parent. Searches for the nearest adult of the same type and pathfinds toward it. Stops when within a close distance. Only active when the mob is a baby (isBaby() == true).
Owner Goals (2)
Goal Description
FollowOwnerGoal Tamed mobs follow their owner. Pathfinds toward the owner when the distance exceeds a threshold. Teleports to the owner if the distance becomes too large (>12 blocks). Disabled when the mob is sitting (SitGoal active).
ControlledByPlayerGoal Allows a player to directly control a mob's movement when riding it (e.g., saddled pig with carrot on a stick). Overrides the normal AI movement, reading the player's input direction and applying it as the mob's movement vector. Speed is boosted briefly when the player uses the carrot on a stick.
Combat Goals (5)
Goal Description
MeleeAttackGoal Chases and melee-attacks the target. Pathfinds toward the target, and when within reach (bounding box overlap), calls doHurtTarget(). Has an attack cooldown (typically 20 ticks) between swings. Includes a speed multiplier when attacking.
ArrowAttackGoal Ranged bow attack behavior used by skeletons. Strafes around the target while maintaining line of sight, then charges and fires an arrow. Attack speed scales with difficulty. The mob switches to melee if the target gets too close.
LeapAtTargetGoal Causes the mob to jump toward its target in a leaping attack. Used by spiders and wolves. Applies a velocity boost on the Y axis and toward the target's position. Typically a short-range engagement initiation.
NearestAttackableTargetGoal Scans for the nearest entity matching a target class (e.g., Player, Villager) and sets it as the mob's attack target. Uses configurable search radius, line-of-sight requirement, and target predicate. The primary targeting goal for most hostile mobs.
NonTameRandomTargetGoal Variant of NearestAttackableTargetGoal that only activates when the mob is not tamed. Used by wolves to hunt sheep and rabbits in their wild (untamed) state. Once tamed, this goal is suppressed.
Targeting Goals (4)
Goal Description
HurtByTargetGoal Sets the attack target to whatever entity last hurt this mob. Provides reactive aggression — mobs fight back when attacked. Optionally alerts nearby mobs of the same type (group aggro), used by pig zombies and wolves.
OwnerHurtByTargetGoal Tamed mobs target entities that hurt their owner. When the owner takes damage, this goal sets the attacker as the pet's target, causing it to defend its owner. Used by tamed wolves.
OwnerHurtTargetGoal Tamed mobs target entities that the owner attacks. When the owner hits an entity, this goal sets that entity as the pet's target, causing it to assist in combat. Used by tamed wolves.
DefendVillageTargetGoal Iron golems target mobs that have recently attacked a villager. Queries the village's aggressor list and targets the nearest one. Only used by VillagerGolem. Provides autonomous village defense.
Interaction Goals (6)
Goal Description
BreedGoal Finds a nearby mate of the same type and moves toward it for breeding. Both animals must be in love mode (isInLove()). When they meet, breedWith() is called to spawn a baby and reset the love timers.
MakeLoveGoal Villager breeding. Two villagers approach each other, and if the village has housing capacity (more doors than villagers), they produce a baby villager. Checks willingness (food items in inventory) before breeding.
TemptGoal Animals follow a player holding a specific food item (wheat for cows/sheep, seeds for chickens, carrots for pigs). The mob pathfinds toward the player at increased speed. Interrupted if the player moves too far away or switches items.
PlayGoal Baby villagers run around and play. Selects random destinations and moves quickly between them. Only active for baby villagers, giving them a lively, childlike behavior pattern.
OfferFlowerGoal Iron golems occasionally offer a poppy (rose) to baby villagers. The golem pathfinds to a baby villager, then holds out a flower for several seconds. Purely cosmetic behavior that adds character.
TakeFlowerGoal Baby villagers approach an iron golem that is offering a flower and "take" it. Complements OfferFlowerGoal on the villager side of the interaction.
Look Goals (3)
Goal Description
LookAtPlayerGoal The mob turns its head to look at a nearby player. Activates when a player is within a configurable range (typically 6–8 blocks). Has a random chance of activating each tick so mobs do not stare constantly. Low priority idle behavior.
LookAtTradingPlayerGoal Villagers look at the player they are currently trading with. Higher priority than generic LookAtPlayerGoal — ensures the villager maintains eye contact during the trade interaction for a natural feel.
RandomLookAroundGoal The mob occasionally turns its head in a random direction when idle. Provides subtle ambient animation so mobs do not appear frozen when standing still. Very low priority.
Door Goals (4)
Goal Description
OpenDoorGoal Opens wooden doors when the mob's path passes through them. Used by villagers to enter and exit buildings. The mob opens the door, walks through, then closes it behind itself. Requires canOpenDoors flag on the navigation.
BreakDoorGoal Zombies break down wooden doors on Hard difficulty. The zombie stands at a door and progressively damages it (visual cracking stages) over several seconds until the door breaks. Only active on Hard difficulty.
DoorInteractGoal Base class for door-related goals. Detects when the mob is adjacent to a door block along its path. Provides shared door-finding logic used by both OpenDoorGoal and BreakDoorGoal.
RestrictOpenDoorGoal Prevents mobs from walking through open doors when they should not. Used to keep certain mobs confined to specific areas (e.g., keep a villager's movement restricted to indoors at night even if the door is open).
Environment Goals (5)
Goal Description
PanicGoal When hurt, the animal runs away at increased speed in a random direction. Activates whenever the mob takes damage. High priority so it overrides normal idle behavior. Used by all passive animals (cows, pigs, sheep, chickens).
AvoidPlayerGoal The mob flees from nearby players. Calculates an escape vector away from the player and pathfinds along it. Used by ocelots (when wild) and villagers (when fleeing from zombies). Configurable avoidance distance and speed.
FleeSunGoal Undead mobs (zombies, skeletons) seek shade when burning in sunlight. Finds the nearest block with a roof overhead and navigates to it. Critical survival behavior for undead mobs during daytime.
RestrictSunGoal Prevents undead mobs from pathfinding into sunlit areas. Modifies the navigation to avoid open-sky blocks, causing the mob to prefer shaded routes. Works in conjunction with FleeSunGoal.
EatTileGoal Sheep eat grass blocks, converting them to dirt and regrowing wool. The sheep lowers its head for a brief eating animation, then the block is changed. Also eats tall grass items. Triggers wool regrowth if the sheep has been sheared.
Special Goals (4)
Goal Description
SwellGoal Creeper-specific explosion goal. When a player is within 3 blocks and has line of sight, the creeper begins swelling (inflating). A 30-tick fuse timer counts up; if it reaches the maximum, the creeper detonates. If the player moves away, the swell timer decreases and the creeper aborts the explosion. The swell progress is synched to clients for the visual inflation effect.
SitGoal Tamed mobs sit when commanded by their owner. The mob becomes stationary and all movement goals are suppressed. Toggled by the owner right-clicking the pet. The sitting state is stored in synched entity data and persists across save/load.
BegGoal Wolf-specific begging behavior. When a player within 8 blocks holds food (bones or meat), the wolf tilts its head. Purely visual — does not affect gameplay beyond the head-tilt animation. The beg state is synched to clients.
OcelotSitOnTileGoal Tamed cats seek out and sit on specific blocks: chests, beds, and active furnaces. The cat pathfinds to the nearest qualifying block and perches on top of it. When sitting on a chest, the chest cannot be opened by players — a deliberate game mechanic.

Pathfinding

The pathfinding system enables mobs to navigate around obstacles, through doors, and across terrain to reach their destinations. It consists of three main classes: the A* algorithm implementation (PathFinder), the navigation manager (PathNavigation), and the path data structure (Path).

PathFinder A* Algorithm

Implements the A* search algorithm for finding optimal paths through the block grid. Takes a start and end position, explores neighboring nodes weighted by traversal cost, and produces the shortest walkable path.

Members:

  • level (LevelSource*) — Reference to the level for block data lookups during pathfinding.
  • openSet (BinaryHeap) — Priority queue of nodes to explore, ordered by estimated total cost (g + h). Uses a binary min-heap for efficient extraction of the lowest-cost node.
  • nodes (map) — Map of all explored nodes keyed by position, preventing duplicate exploration and enabling path reconstruction.
  • neighbors — Reusable buffer for neighboring nodes during expansion. Avoids repeated allocation during the search loop.

Key Methods:

  • findPath(Entity from, Entity to, float maxDist) — Finds a path from one entity to another. The maxDist parameter limits the search radius to prevent expensive long-distance searches.
  • findPath(Entity, int x, int y, int z, float maxDist) — Finds a path from an entity to a specific block coordinate. Used when the target is a position rather than an entity.
  • getNeighbors() — Returns the walkable neighbors of the current node. Checks all 6 cardinal directions (and diagonals on the horizontal plane) for traversability.
  • isFree() — Checks whether a given block position is free for the mob to occupy. Considers block solidity, the mob's bounding box dimensions, and water/lava avoidance settings.

Node Types

Constant Value Description
WALKABLE 2 The node is on solid ground and the mob can walk through it freely. Standard traversal cost.
OPEN 1 The node is passable but not on solid ground (e.g., air above a gap). The mob would need to fall or jump to reach it.
BLOCKED 0 The node is occupied by a solid block and cannot be traversed. The pathfinder routes around blocked nodes.
WATER -1 The node is in water. Mobs with avoidWater enabled will not path through these nodes. Aquatic mobs treat these as normal walkable nodes.
LAVA -2 The node is in lava. Almost all mobs avoid lava nodes. Fire-immune mobs (blazes, pig zombies) may traverse these.
FENCE -3 The node is on top of a fence or wall. Has a 1.5-block effective height, making it impassable to standard-height mobs. Prevents mobs from pathing over fences.
TRAP -4 The node contains a hazard (cactus, fire, etc.). The pathfinder adds a large extra cost to traversing traps, making mobs avoid them unless no other path exists.
PathNavigation Navigation Manager

Manages mob pathfinding at a high level. Creates paths via PathFinder, follows them tick-by-tick, and handles path invalidation when blocks change or the target moves. Each mob has its own PathNavigation instance configured with mob-specific settings.

Members:

  • path (Path*) — The currently active path being followed. nullptr when the mob has no destination.
  • speed — Movement speed multiplier applied when following the path. Set by the goal that initiated the path.
  • maxDist — Maximum pathfinding search distance. Limits A* expansion to prevent expensive searches. Typically 16–32 blocks.
  • canPassDoors — Whether the mob can path through closed door blocks (treats doors as passable).
  • canOpenDoors — Whether the mob can actually open doors when it reaches them. Requires canPassDoors to be true.
  • avoidSun — Whether the mob avoids sunlit blocks (used by undead mobs to stay in shade).
  • avoidWater — Whether the mob avoids water blocks during pathfinding.
  • canFloat — Whether the mob can swim (path through water surface) rather than sinking.

Key Methods:

  • createPath(int x, int y, int z) — Creates a new path to the given block coordinates using the A* pathfinder.
  • createPath(Mob target) — Creates a path to a target mob's current position.
  • moveTo() — Begins following the current path. Sets the mob's movement control to walk toward the next path node.
  • tick() — Per-tick navigation update. Advances along the path, checks for obstacles, and recalculates the path if the current one becomes invalid.
  • isDone() — Returns true if the mob has reached the end of its path or has no active path.
  • setAvoidWater(bool) — Enables or disables water avoidance for this mob's pathfinding.
  • setCanOpenDoors(bool) — Enables or disables the ability to open doors during pathfinding.
Path Path Data

Represents a calculated path as an ordered sequence of nodes from start to destination. Produced by PathFinder and consumed by PathNavigation. The path is immutable once created; if the environment changes, a new path must be calculated.

Members:

  • nodes (NodeArray) — Ordered array of path nodes from start to end. Each node contains a block position (x, y, z) and the node type.
  • index — Current position along the path. Incremented as the mob reaches each successive node.
  • length — Total number of nodes in the path.

Key Methods:

  • next() — Advances to the next node in the path. Increments index.
  • isDone() — Returns true if index >= length, meaning the mob has traversed the entire path.
  • get(int i) — Returns the node at position i in the path array.
  • currentPos(Entity) — Returns the world position of the current node, adjusted to the entity's Y offset.
  • endsIn(Pos) — Returns true if the path's final node matches the given block position. Used to check if a path actually reaches the intended destination.

Control Systems

Control systems act as the interface between AI goals and the mob's physical movement. Goals set desired states (where to look, where to move, whether to jump) and the controls translate those desires into smooth, physically plausible motion each tick.

LookControl Head Rotation

Controls the mob's head rotation. Goals call setLookAt() to specify a target position, and the control smoothly interpolates the head rotation toward that position over multiple ticks.

Members: wantedX, wantedY, wantedZ (target look position), yMax (maximum yaw rotation per tick), xMax (maximum pitch rotation per tick).

Behavior: tick() calculates the angle difference between the current and wanted look direction, then applies a clamped rotation step. This produces smooth, natural-looking head movement rather than instant snapping.

MoveControl Movement

Controls the mob's ground and swimming movement. Goals call setWantedPosition() with a target coordinate and speed, and the control applies the appropriate movement forces each tick.

Members: wantedX, wantedY, wantedZ (target position), speed (movement speed multiplier).

Behavior: tick() calculates the direction toward the wanted position, rotates the mob to face that direction, and applies forward movement at the configured speed. Handles both ground walking and swimming movement.

JumpControl Jumping

Manages jump requests. Goals call jump() to request a jump, and the control executes it on the next tick when the mob is on the ground.

Methods: jump() — Sets the jump flag. tick() — If the jump flag is set and the mob is onGround, applies an upward velocity impulse and clears the flag.

BodyControl Body Posture

Controls body rotation relative to the head. Prevents the body from rotating to unrealistic angles when the head is turned far to the side.

Behavior: clamp() checks the angle between the head yaw and body yaw. If the difference exceeds the maximum allowed angle, the body is gradually rotated to follow the head, producing a natural turning animation rather than a disjointed appearance.

Sensing Vision System

Provides cached line-of-sight checks for the mob's AI system. Multiple goals can query visibility of the same entity without duplicating expensive raycasts.

Methods: canSee(Entity*) — Returns whether this mob has line of sight to the given entity. Results are cached per tick so the first call performs a raycast and subsequent calls return the cached result.

Behavior: Tracks two lists internally — seen entities and unseen entities — which are cleared and rebuilt each tick. The raycast checks for solid block obstruction between the mob's eyes and the target's eye height.

Other Entity Types

Beyond mobs and players, the entity system includes several non-living entity types that handle projectiles, vehicles, items, decorations, weather effects, and mechanical blocks.

Arrow Projectile

Projectile entity fired by bows and skeleton mobs. Travels in a ballistic arc with gravity. On collision with a mob, applies damage based on velocity magnitude. On collision with a block, sticks in place and can be picked up by the player after a short delay. Supports enchantment effects (fire, infinity, power) applied at launch time. Tracks the shooter entity to prevent self-damage.

Boat Vehicle

Rideable vehicle entity for water transport. Floats on water surfaces and can be steered by the riding player. Has its own collision and physics system tuned for water interaction (buoyancy, drag). Can be placed on land but moves very slowly. Breaks into planks and sticks if it hits a block at sufficient speed or takes enough damage.

ItemEntity Dropped Item

Represents an item dropped in the world. Has full physics (gravity, block collision, water floating) and a bobbing/spinning visual animation. Nearby matching item entities merge stacks automatically. Picked up by players who walk over them (added to inventory). Despawns after 5 minutes (6000 ticks) to prevent accumulation. Destroyed by fire, lava, explosions, and cacti.

HangingEntity Wall Decoration

Base class for entities that hang on walls: Painting and ItemFrame. Attached to a specific block face and direction. Checks each tick that the supporting block still exists; drops as an item if the block is removed. Has a specialized flat bounding box oriented along the wall surface.

GlobalEntity Weather Effect

Lightning bolt entity. Exists briefly in the world, dealing damage to nearby entities, setting fire to blocks, and producing the lightning visual/audio effect. Can convert pigs to pig zombies, charge creepers, and transform villagers. Rendered globally (visible from any distance) rather than being culled by normal entity render distance.

ItemFrame Display Entity

Extends HangingEntity. Displays an item on a wall surface. The contained item can be rotated in 8 orientations by right-clicking. When an item frame contains a map, the map is rendered on the wall at full scale. Drops both the frame and the contained item when broken.

PistonPieceEntity Mechanical

Temporary entity representing a block being pushed or pulled by a piston. Exists for exactly 2 ticks (one piston extension cycle) while the block visually slides from its old position to its new position. Handles the rendering of the moving block and its collision during the transition. Replaced by a normal block at the destination once the piston cycle completes.
