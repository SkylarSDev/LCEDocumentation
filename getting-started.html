Getting Started

This guide covers building the LCEMP project from source. It walks you through the required development tools, game assets, third-party libraries, build configuration, and launch arguments needed to compile and run the Minecraft Legacy Console Edition Windows 64-bit client.

Prerequisites

Before building the project, ensure your development environment meets the following requirements:

Development Environment Windows 64-bit
  • Visual Studio 2012 or newer — The solution and project files target the MSVC toolchain. Visual Studio 2012 is the minimum supported version; newer versions (2015, 2017, 2019, 2022) are also compatible.
  • CMake 3.10+ — Used to generate build files if you prefer CMake over the Visual Studio solution directly.
  • C++11 compatible compiler — The codebase relies on C++11 features such as auto, range-based for loops, nullptr, and std::shared_ptr.
  • DirectX 11 SDK — Required for the Windows rendering backend. Typically included with the Windows SDK in Visual Studio 2012+.
  • Windows 64-bit development environment — The build targets x64 exclusively. A 32-bit host is not supported.

Note: The original codebase was developed with Visual Studio 2012. While newer compilers work, you may encounter warnings about deprecated C++ features or SDK differences. The recommended approach is to use Visual Studio 2019 or 2022 with the v110 or v140+ platform toolset.

Required Assets

The LCEMP source code does not include proprietary game assets. You must supply the following directories from a legitimate copy of Minecraft Legacy Console Edition. Place them in the correct paths relative to the project root.

Directory Contents
Minecraft.Client/music/ Binka audio format music files (.binka). These are the in-game music tracks encoded in RAD Game Tools' Binka format.
Minecraft.Client/Common/Media/ SWF UI files for multiple resolutions: 480p, 720p, 1080p, and PS Vita resolution variants. These Flash-based assets drive the Iggy/Scaleform UI system.
Minecraft.Client/Common/res/ Game textures and resources, including block textures, entity textures, GUI elements, fonts, and shader files.
Minecraft.Client/Common/DummyTexturePack/ Fallback texture pack used when no other texture pack is loaded. Provides a minimal set of textures to prevent rendering errors.
Minecraft.Client/DurangoMedia/ Xbox One (Durango) media assets, including platform-specific UI layouts, splash screens, and certification assets.
Minecraft.Client/OrbisMedia/ PS4 (Orbis) media assets, including PlayStation-specific button prompts, trophy images, and store assets.
Minecraft.Client/PS3Media/ PS3 media assets, including platform-specific textures, icons, and certification-required resources.
Minecraft.Client/PSVitaMedia/ PS Vita media assets, including scaled-down textures and Vita-specific UI resources optimized for the handheld display.
Minecraft.Client/Windows64Media/ Windows PC media assets. This is the primary media directory used when building and running the Windows 64-bit target.
Minecraft.Client/redist64/ Redistributable runtime files required for the 64-bit build, including Visual C++ runtime DLLs and other dependencies.

Required Libraries

The project depends on several proprietary middleware libraries. These must be obtained separately and placed in the correct directories. Each library ships per-platform builds.

4J Studios Libraries

Proprietary libraries developed by 4J Studios that provide platform-abstracted input, storage, profiling, and rendering services.

Directory Platform Libraries Included
4JLibs/Windows64/ Windows 64-bit 4J_Input, 4J_Storage, 4J_Profile, 4J_Render_PC
4JLibs/Xbox/ Xbox 360 4J_Input, 4J_Storage, 4J_Profile, 4J_Render_PC
4JLibs/Durango/ Xbox One 4J_Input, 4J_Storage, 4J_Profile, 4J_Render_PC
4JLibs/Orbis/ PS4 4J_Input, 4J_Storage, 4J_Profile, 4J_Render_PC
4JLibs/PS3/ PS3 4J_Input, 4J_Storage, 4J_Profile, 4J_Render_PC
4JLibs/PSVita/ PS Vita 4J_Input, 4J_Storage, 4J_Profile, 4J_Render_PC

Miles Sound System

RAD Game Tools' Miles Sound System provides audio playback, 3D positional sound, and streaming capabilities.

Miles/ Audio Middleware

Per-platform directories containing mss64.lib (Windows 64-bit static import library). The Miles SDK provides the audio backend used for all sound effects, ambient audio, and music playback. Platform-specific variants exist for each target console.

Iggy UI Middleware

Iggy (Scaleform) provides Flash/SWF-based UI rendering used for all menu screens and HUD elements.

Iggy/ UI Middleware

Per-platform directories containing iggy_w64.lib (Windows 64-bit static import library). Iggy renders SWF assets at runtime and bridges ActionScript calls to the C++ game code via the IggyInterface class.

Sentient Middleware

Sentient provides additional middleware services used by the engine.

Sentient/ Middleware

Per-platform directories containing Sentient middleware libraries. Like the other middleware dependencies, platform-specific builds must be placed in the corresponding subdirectories.

Required Files

In addition to the asset directories and library folders, the following individual files are required for a successful build and runtime execution.

Build-Time Files

File Description
Minecraft.Client/xbox/MinecraftWindows.rc Windows resource file that embeds version information, application manifest, and icon resources into the compiled executable.
Minecraft.Client/xbox/MinecraftWindows.ico Application icon displayed in the taskbar, window title bar, and file explorer. Standard multi-resolution .ico format.
Minecraft.Client/xbox/small.ico Small application icon (typically 16x16 and 32x32) used for taskbar and ALT+TAB displays.

Runtime DLLs

These DLLs must be present in the output directory alongside the built executable.

File Description
x64/Debug/iggy_w64.dll Iggy UI runtime DLL. Provides the Flash/SWF rendering engine at runtime. Must be in the same directory as the executable.
x64/Debug/mss64.dll Miles Sound System runtime DLL. Required for all audio playback functionality. Must be in the same directory as the executable.

Important: If the runtime DLLs are missing, the application will fail to launch with a system error about missing DLL dependencies. Ensure both iggy_w64.dll and mss64.dll are copied to your build output directory.

Build Instructions

Follow these steps to build the LCEMP project from source. The build produces two targets: a static library (MinecraftWorld) and the final executable (MinecraftClient).

Step-by-Step

  1. Obtain the required assets

    Gather all game textures, music files, and platform media directories listed in the Required Assets section. Place them in the expected paths relative to the project root.

  2. Replace source folders

    Replace your Minecraft.Client and Minecraft.World source folders with the ones from this repository. The LCEMP modifications live in these two directories.

  3. Build using CMake or Visual Studio

    You can build using either method:

    Option A: Visual Studio Solution

    Open MinecraftConsoles.sln in Visual Studio. Select the x64 platform and your desired configuration (Debug or Release), then build the solution.

    1. Open MinecraftConsoles.sln
2. Set platform to x64
3. Set configuration to Debug or Release
4. Build > Build Solution  (Ctrl+Shift+B)
    Option B: CMake

    Generate the build files using CMake, then compile:

    mkdir build
cd build
cmake .. -G "Visual Studio 17 2022" -A x64
cmake --build . --config Debug

    Adjust the generator string for your installed Visual Studio version.

  4. Run with optional launch arguments

    After a successful build, the executable is output to x64/Debug/ (or x64/Release/). See the Launch Arguments section for runtime options.

Build Targets

Target Type Description
MinecraftWorld Static Library Compiles all Minecraft.World source files into a static library (.lib). Contains entities, tiles, items, networking, world generation, storage, and gameplay logic.
MinecraftClient Win32 Executable Links MinecraftWorld.lib together with the Minecraft.Client source files and all third-party libraries to produce the final Minecraft.Client.exe executable.

Preprocessor Defines

The following preprocessor macros control build-time feature flags and platform targeting. These are typically set in the project properties or CMake configuration.

Define Description
_LARGE_WORLDS Enables large world support. When defined, the engine uses expanded coordinate ranges and larger chunk loading distances, removing the default console world-size limitations.
_DEBUG_MENUS_ENABLED Enables debug menus and developer overlays. Provides access to in-game debug tools such as chunk visualization, entity inspection, network statistics, and performance profiling.
_WINDOWS64 Windows 64-bit target platform flag. Must be defined when building for the Windows x64 platform. Controls platform-specific code paths throughout the engine.
Setting Defines in Visual Studio

In Visual Studio, navigate to Project Properties > C/C++ > Preprocessor > Preprocessor Definitions and add the defines separated by semicolons:

_WINDOWS64;_LARGE_WORLDS;_DEBUG_MENUS_ENABLED

For Release builds, you may want to omit _DEBUG_MENUS_ENABLED to disable debug overlays.

Launch Arguments

The client executable supports several command-line arguments to configure networking and player identity at launch. These are especially useful for LAN multiplayer testing.

Argument Usage Description
-name -name <username> Sets your in-game username. This is the name displayed above your player character and in the player list. If not specified, a default name is used.
-ip -ip <targetip> Manually connect to a specific IP address. Use this if LAN discovery fails to find the host or if you are connecting across subnets.
-port -port <targetport> Override the default port number for the connection. The default port is 25565. Useful when running multiple instances or when port forwarding is configured on a non-standard port.

Example

Launch the client with a custom username and connect directly to a LAN host:

Minecraft.Client.exe -name Steve -ip 192.168.0.25 -port 25565
Running from Visual Studio

To pass launch arguments when debugging in Visual Studio, go to Project Properties > Debugging > Command Arguments and enter:

-name Steve -ip 192.168.0.25 -port 25565

Linked Libraries

The MinecraftClient target links against the following libraries at build time. These must all be available in your library search paths for the linker to succeed.

Library Source Description
4J_Input 4J Proprietary Platform-abstracted input handling. Provides a unified API for keyboard, mouse, and gamepad input across all supported platforms.
4J_Storage 4J Proprietary Platform-abstracted save data management. Handles reading and writing save files through each platform's native storage APIs.
4J_Profile 4J Proprietary Player profile and user account management. Interfaces with platform-specific user identity systems.
4J_Render_PC 4J Proprietary PC-specific rendering abstraction layer built on top of DirectX 11. Provides the low-level graphics interface used by the engine's renderer.
iggy_w64.lib UI Middleware Iggy/Scaleform Flash UI engine. Renders SWF-based menus and HUD elements, and provides the ActionScript-to-C++ bridge for UI interactions.
mss64.lib Audio Middleware Miles Sound System by RAD Game Tools. Provides audio mixing, 3D positional sound, streaming music playback, and Binka format decoding.
d3d11.lib Windows SDK DirectX 11 graphics API. The core rendering API used for all GPU operations: shader compilation, buffer management, draw calls, and swap chain presentation.
XInput9_1_0.lib Windows SDK XInput controller input library. Provides Xbox-compatible gamepad support on Windows, including analog sticks, triggers, and rumble feedback.
Linker Configuration in Visual Studio

In Visual Studio, navigate to Project Properties > Linker > Input > Additional Dependencies and ensure the following are listed:

4J_Input.lib
4J_Storage.lib
4J_Profile.lib
4J_Render_PC.lib
iggy_w64.lib
mss64.lib
d3d11.lib
XInput9_1_0.lib

Also verify that Linker > General > Additional Library Directories includes the paths to your 4JLibs/Windows64/, Miles/, and Iggy/ directories.
